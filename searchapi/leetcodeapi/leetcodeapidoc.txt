LEETCODE API CHEAT SHEET

Topic -> Description -> Example

1. LeetCode API
-> Online coding platform API
-> Problems, Contests, Interviews, Discussions

2. Base URL
-> Root endpoint for all services
-> api.leetcode.com

3. Main Services
-> User, Problem, Contest, Interview, Discuss
-> Core modules of LeetCode

--------------------------------------------------
USER SERVICE
--------------------------------------------------

4. Register User
-> Create new account
-> POST /users/register

5. Login User
-> Authenticate user
-> POST /users/login

6. User Profile
-> Fetch user information
-> GET /users/profile/{userId}

7. User Data
-> Stores user metadata
-> Name, Email, Username, Ranking

--------------------------------------------------
PROBLEM SERVICE
--------------------------------------------------

8. List Problems
-> Fetch coding problems
-> GET /problems

9. Search Problems
-> Search by topic/category
-> GET /problems?query=Graph

10. Evaluate Code
-> Run solution against test cases
-> POST /problems/evaluate

11. Problem Metadata
-> Stores question details
-> Difficulty, Tags, Acceptance Rate

12. Code Execution Service
-> Executes submitted code
-> Docker Containers/Sandbox

13. Evaluation Result
-> Returns correctness result
-> Passed 15/15 Test Cases

--------------------------------------------------
CONTEST SERVICE
--------------------------------------------------

14. Register Contest
-> Enroll user into contest
-> POST /contest/register

15. Submit Code
-> Submit contest solution
-> POST /contest/submit

16. Contest History
-> View previous contests
-> GET /contest/history

17. Leaderboard
-> Contest rankings
-> GET /contest/leaderboard/{contestId}

18. Contest Workflow
-> Submit → Evaluate → Rank
-> Updates leaderboard automatically

19. Plagiarism Service
-> Detect copied code
-> Compare submissions

20. Why 202 Accepted?
-> Evaluation happens asynchronously
-> Code queued for judging

--------------------------------------------------

23. Live Coding

--------------------------------------------------
ARCHITECTURE
--------------------------------------------------

31. REST
-> Client ↔ API Gateway communication
-> GET /problems

32. GraphQL
-> API Gateway ↔ Backend Services
-> Fetch user + contest + problem data in one query

33. Event Driven
-> Notifications and async processing
-> Contest evaluation updates

34. Why GraphQL?
-> Single request for multiple services
-> Reduces over-fetching

35. Why REST for Clients?
-> Simple and widely supported
-> Easy CRUD operations

--------------------------------------------------
IMPORTANT STATUS CODES
--------------------------------------------------

50. 200 OK
-> Successful fetch/update

51. 201 Created
-> Resource created
-> Register User, Register Contest

52. 202 Accepted
-> Processing asynchronously
-> Submit Contest Code

53. 400 Bad Request
-> Invalid input

54. 403 Forbidden
-> No permission

55. 404 Not Found
-> Resource missing

56. 500 Internal Server Error
-> Server failure

--------------------------------------------------
INTERVIEW QUESTIONS
--------------------------------------------------

57. Difference: Evaluate vs Submit?

Evaluate
-> Practice Mode
-> Immediate Result
-> 200 OK

Submit
-> Contest Mode
-> Async Processing
-> 202 Accepted

58. Why GraphQL instead of REST internally?

-> Multiple services involved
-> Single query can fetch all required data
-> Lower latency

59. Why Pub/Sub?

-> Contest evaluation, notifications, interview events
-> Loose coupling between services

60. Why Containers?

-> Secure code execution
-> Isolation and scalability
